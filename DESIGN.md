## Problem Statement

Teams shipping LLM features can't answer "who is burning tokens?" Spend hides in long prompts, retries, agent loops, and tool calls; a few expensive calls dominate; blowups appear only on the monthly invoice. Model-API spend grew $3.5B → $8.4B (late-2024 → mid-2025). The two incumbent answers are newly weakened: Helicone was acquired (March 2026, maintenance mode, 80+ outages/yr, 50–80ms added latency) and LiteLLM had a supply-chain compromise (March 2026), 1000+ open issues, Python-GIL throughput decay (1.7–4x under load), and its Postgres logging sits in the request path.

## What Makes This Cool

- Change `base_url` in your OpenAI/Anthropic SDK → every request is metered, attributed, and budget-enforced. Zero code changes.
- Live per-tenant / per-feature cost dashboard (SSE) — the 30-second demo.
- A tenant hits its monthly budget cap mid-demo and the gateway blocks it atomically (Redis), across replicas.
- One static Go binary. `docker compose up` → working gateway in 60 seconds. Benchmark table in the README: p95 added latency < 5ms, req/s under load.

## Premises (agreed 2026-07-05)

1. Target user: solo devs / small teams needing per-tenant cost attribution + self-hosting without SaaS lock-in.
2. Differentiators map 1:1 to documented competitor pains: Go single binary (vs LiteLLM GIL + Prisma setup pain); metering writes OUT of the hot path via async batching (vs LiteLLM DB-in-request-path); <5ms p95 overhead target (vs Helicone 50–80ms); per-tenant attribution first-class; no eval/prompt-management bundling.
3. Timing window is real: Helicone maintenance mode + LiteLLM trust damage.
4. v1 providers: OpenAI + Anthropic only. SSE streaming passthrough with token counting is mandatory.
5. Distribution: GitHub Releases binary + Docker image + docker-compose (Postgres + Redis included). Repo in ~/Projects/spendgate.

## Approaches Considered

### Approach A: Minimal viable (S, Low risk)
Single binary, proxy + async metering → Postgres, polling dashboard, approximate in-process budget counters, no Redis. Rejected: approximate budgets under multi-replica; weaker systems story for the portfolio.

### Approach B: Ideal architecture (M, Med risk) — CHOSEN
Proxy with SSE streaming passthrough + token counting; Redis atomic per-tenant budget/rate counters (exact, multi-replica safe); batched async Postgres writes; live SSE dashboard; per-tenant API keys; load-test numbers in README.

### Approach C: Go middleware library-first (M, Med risk)
Embeddable `net/http` middleware, binary secondary. Rejected for v1: narrows audience to Go shops, weaker demo. Kept as a possible v2 packaging.

## Recommended Approach

**B.** Architecture:

```
Client SDK (base_url swapped)
   │  Authorization: Bearer tb_<tenant-key>   X-Spendgate-Feature: <tag>
   ▼
spendgate proxy (Go, single binary)
   ├── auth: tenant key lookup (in-memory cache, 30s TTL, Postgres source of truth;
   │     revocation takes effect within TTL — accepted for v1)
   ├── budget check: Redis atomic counters (Lua/INCR). Over budget → 429 with
   │     JSON body naming the exceeded budget. Redis down → per-tenant flag:
   │     fail-open (forward unmetered, log warning) or fail-closed (503)
   ├── forward to provider (OpenAI | Anthropic); SSE passthrough streamed as-is.
   │     Provider errors (4xx/5xx/429/timeouts/mid-stream disconnects) pass through
   │     unmodified — the gateway never retries; retry policy belongs to the client
   ├── token/cost extraction: usage from response body / final SSE usage frame;
   │     fallback if usage absent: chars/4 heuristic, row marked estimated=true
   │     (no tokenizer dependency in v1)
   └── metering event → in-process buffered channel
              │ (batch insert, N rows or T ms, whichever first)
              ▼
        Postgres (requests, tenants, budgets, price table)
              ▲
   dashboard (embedded in same binary): SSE live spend, per-tenant/feature breakdown
```

Key decisions:
- **Hot path touches only Redis** (single round-trip budget check); Postgres is async-batch only. This is the direct answer to LiteLLM's documented bottleneck.
- **Provider-reported usage is the source of truth** for token counts (both OpenAI and Anthropic emit usage in final stream frames); local estimation is fallback only. Avoids tokenizer drift.
- **Price table is data, not code**: Postgres table seeded from a checked-in YAML file at startup (upsert); updating prices = edit YAML + restart, no admin API in v1.
- **Metering buffer is lossy on crash by design** (kill -9 loses ≤1 batch, i.e. ≤N rows or T ms of data). Graceful shutdown flushes. Accepted for v1: this is metering, not billing-grade ledger; documented in README.
- **Tenant/key creation**: CLI subcommand (`spendgate create-tenant <name>`) inserts the row and prints the key once. No HTTP admin API in v1. Rotation/revocation: delete/insert row in Postgres; effective within cache TTL (30s).
- **Rate limiting is a non-goal for v1** — budget caps only. (Per-tenant RPS caps are a natural v2 on the same Redis counters.)
- **Dashboard**: read-only, no auth in v1, binds to localhost by default (documented; put a reverse proxy in front for shared access).
- **Fail-open vs fail-closed on Redis outage is a per-tenant config flag** — the kind of tradeoff hiring managers ask about.
- No eval, no prompt management, no provider abstraction layer beyond the two supported APIs. Deliberate non-goals, stated in README.

## Success Criteria

- `docker compose up` to first metered request in < 60s, documented.
- p95 gateway overhead < 5ms at 500 concurrent req/s (vegeta/k6 numbers in README).
- Streaming responses content-identical to direct provider calls at the SSE-event level (chunk-boundary-tolerant golden tests).
- Budget cap enforced exactly under concurrent load across 2 replicas (integration test).
- Dashboard shows live spend per tenant/feature within 2s of request completion.
- README includes benchmark table + a "Design decisions" section; ≥1 blog-style writeup.
